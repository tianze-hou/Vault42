# SCHEMA.md — Knowledge Base Schema

> 本文件是 AI Agent 的行为配置文件（Schema）。
> 每次会话开始时，AI 应先阅读本文件，再阅读 `Wiki/index.md`，再开始工作。

---

## 自定义配置

> 以下配置由初始化时设置，如需修改请直接编辑本部分。

```yaml
# 知识库主题（可选，如：机器学习、产品管理、个人成长）
# 默认为 general（通用知识库）
# 主题：general

# 助手人设风格（可选，如：专业严谨、轻松闲聊、简洁干练）
# 默认为 简洁干练
# 风格：简洁干练
```

## 目录结构

```
\
├── SCHEMA.md              ← 本文件：Schema 配置
├── README.md              ← 使用说明
├── Raw/                   ← 原始资料层（摄取后归档）
│   ├── YYYY-MM/           ← 按月归档
│   └── assets/            ← 图片等附件
└── Wiki/                  ← 知识库层（AI 全权维护）
    ├── index.md           ← 内容索引
    ├── log.md             ← 操作日志（仅追加）
    ├── 概念/              ← 概念页：原理、方法、术语
    ├── 实体/              ← 实体页：人、公司、产品、工具
    ├── 来源/              ← 来源摘要页：每篇资料的提炼
    └── 对比/              ← 对比分析页：跨来源横向综合
```

### 层级规则

| 层级 | 目录 | 谁可以修改 |
|------|------|-----------|
| 原始资料 | `Raw/` | **只有用户**放入新文件，**AI** 摄取后归档至此 |
| 知识库 | `Wiki/` | **只有 AI**，用户可读 |
| Schema | `SCHEMA.md` | 用户和 AI 共同演进 |

## Wiki 页面约定

### Frontmatter（YAML）

所有 Wiki 页面都应有以下 frontmatter：

```yaml
---
type: entity | concept | source | comparison
tags: [标签1, 标签2]
sources: [来源文件名或URL]
created: YYYY-MM-DD
updated: YYYY-MM-DD
---
```

### 页面类型

| 类型 | 目录 | 命名格式 | 说明 |
|------|------|----------|------|
| `entity` | `Wiki/实体/` | `实体名.md` | 具体的人、组织、产品 |
| `concept` | `Wiki/概念/` | 概念名.md | 原理、方法、术语 |
| `source` | `Wiki/来源/` | `YYYY-MM-DD 标题.md` | 原始资料的摘要 |
| `comparison` | `Wiki/对比/` | `对比主题.md` | 跨来源对比分析 |

### 交叉引用规则

- 页面间用 Wiki-link：`[[页面名]]`
- 每个提到的实体/概念都应链接到对应页面
- 如果对应页面不存在，先创建再引用
- 来源摘要页顶部必须链接到对应 Raw 文件：`[[Raw/文件名]]`

### 创建时机规则（自下而上原则）

**核心思想**：Wiki 里的概念必须从多篇来源中自然浮现，不是对单篇文章的过度抽象。

| 页面类型 | 创建条件 |
|----------|----------|
| 实体页 | 同一实体在 **≥2 篇不同来源** 中被明确提及 |
| 概念页 | 同一概念在 **≥2 篇不同来源** 中被讨论 |
| 对比页 | 对比主题已在 **≥2 篇来源** 中出现不同立场或数据 |
| 来源摘要页 | **始终创建**，每篇原始资料都对应一个 |

**具体流程**：

1. **第 1 篇来源**：在来源摘要页列出该概念（但**不创建**独立页）
2. **第 2 篇来源**：检查是否已有来源页提到过该概念 → **是** → 创建；**否** → 继续等待
3. **跨来源识别**：读新文章时，主动回溯已有来源摘要，发现重叠 → 触发建页

## 工作流

### 1. Ingest（摄取新资料）

当用户说"摄取"、"处理"、"存入"时：

1. **扫描** `Raw/` 根目录下所有 `.md` 文件
2. **阅读** 新文件内容（含图片）
3. **讨论** 与用户确认关键要点
4. **创建** `Wiki/来源/YYYY-MM-DD 标题.md` 摘要页
5. **更新** 相关的实体页（不存在则创建）
6. **更新** 相关的概念页（不存在则创建）
7. **更新** `Wiki/index.md`
8. **追加** `Wiki/log.md`
9. **归档** 原始文件到 `Raw/YYYY-MM/`

> 归档后 Obsidian 会自动更新正文中 `[[...]]` wikilink，但 frontmatter 的 `sources:` 字段是纯文本，不会自动更新——这是预期行为。

### 2. Query（查询）

1. **阅读** `Wiki/index.md` 找到相关页面
2. **深入** 阅读相关页面及其链接
3. **综合** 生成回答，标注来源页面
4. **归档**（可选）：如果回答有价值，写入 `Wiki/对比/`
5. **追加** `Wiki/log.md`

### 3. Lint（健康检查）

当用户说"检查 Wiki"、"lint"时：

- [ ] 有无**互相矛盾**的说法
- [ ] 无**孤儿页面**（没有任何入链）
- [ ] 有无**提到但缺页**的实体/概念
- [ ] 有无**过时信息**（被新资料推翻但未更新）
- [ ] `index.md` 是否与实际文件同步
- [ ] 有无值得新建的汇总/对比页

## Index 维护规则

`Wiki/index.md` 的每一行格式：

```markdown
| [[Wiki/来源/2026-04-21 标题]] | 一句话简介 | 2026-04-21 |
```

---

## Log 格式

```markdown
## [YYYY-MM-DD] 操作类型 | 标题

- 操作说明 bullet 1
```

操作类型：`init` / `ingest` / `query` / `lint` / `update`

---

## 图片处理

- 原始资料中的图片应下载到 `Raw/assets/` 目录
- Wiki 页面引用图片用：`![[assets/图片名.png]]`

---

## 语言规则

- 目录名：英文（Raw, Wiki, assets）
- 内容：中文为主
- 保持简洁，避免长难句
